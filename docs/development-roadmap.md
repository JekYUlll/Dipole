# Dipole 开发路线图

## 1. 文档目的

这份路线图用于把 `Dipole` 的开发顺序、阶段目标和阶段后的重构任务明确下来，方便后续持续推进。

当前项目定位仍然是：

- 以 `KamaChat` 为主参考的学习型 IM 项目
- 以 `im-server` 校正模块边界和演进方式
- 以模块化单体作为当前阶段的落地形态

同时明确两件事：

- `AI` 功能是必做主线
- `Cgo` 高性能模块是必做主线

这两部分会在 IM 核心链路稳定后正式接入，前面的阶段需要提前为它们预留边界。

---

## 2. 总体开发顺序

推荐按下面顺序推进：

1. 基线收口与测试基建
2. 用户与认证
3. 消息与 WebSocket
4. 会话
5. 联系人与关系链路
6. 群组
7. AI 能力接入
8. Cgo 高性能模块接入
9. 工程化补强

其中 `1-6` 是 IM 核心主线，`7-8` 是明确要交付的增强主线，`9` 用于把项目打磨到更适合展示和继续扩展的状态。

---

## 3. 贯穿全程的开发原则

- 基于 test 进行开发，每个阶段都要补对应的单元测试和接口测试。
- 每做完一个功能块，就做一次小规模重构，不把债务一路堆到最后。
- 先保证单体内边界清晰，再考虑是否需要拆分进程。
- 纯 Go 版本优先保证正确性，性能优化建立在 benchmark 和 profiling 之上。
- AI 与 Cgo 在前期先预留接口和扩展点，中后期做正式实现。

---

## 4. 阶段路线

### Phase 0：基线收口与测试基建

当前仓库已经有 `auth/user` 雏形，这一阶段的目标是把基础盘子收稳。

开发内容：

- 梳理现有 `user/auth` 代码路径
- 补最小测试基建
- 固化本地开发启动方式
- 整理统一错误码与统一响应结构
- 让数据库迁移与配置加载更可控

交付清单：

- `go test ./...` 下具备基础测试文件
- `auth service`、`user service`、`middleware`、`handler` 的最小测试覆盖
- 文档化本地启动步骤和依赖
- 稳定的 `docker-compose` 联调方式

阶段后重构任务：

- 把 `server.New()` 里的对象装配逐步收口到 `bootstrap`
- 把 `store.DB`、`store.RDB` 这类全局变量逐步包进更清晰的初始化边界
- 为 repository 抽出更稳定的接口，方便 service 测试替身接入
- 把 `internal/handler/http` 中的公共响应逻辑抽得更明确

完成标志：

- 当前 `auth/user` 链路可测试、可启动、可联调
- 新增功能前不需要继续返工基础脚手架

### Phase 1：用户与认证

这是当前最应该优先做稳的业务阶段。

开发内容：

- 完成注册、登录、登出
- 完成 `GET /users/me`
- 完成 `GET /users/:uuid`
- 完成 `PATCH /users/:uuid/profile`
- 补齐昵称、头像、邮箱等资料修改能力
- 为封禁、管理员、状态字段预留后续入口

交付清单：

- 用户资料修改接口
- 更完整的 `User` 模型与 DTO
- 密码哈希、token 生命周期、登录态校验
- 用户资料接口测试、服务测试、仓储集成测试

阶段后重构任务：

- 将 `auth` 与 `user` 从当前 `service/repository/handler` 过渡到更清晰的模块边界
- 把输入输出 DTO 与数据库模型解耦
- 为 token、password、uuid 生成等能力建立独立小组件
- 给用户状态、管理员能力增加更清楚的领域常量与方法

完成标志：

- 用户体系形成完整闭环
- 后续联系人、会话、消息都可以稳定依赖 `user` 域

### Phase 2：消息与 WebSocket

这是 IM 项目的第一个核心展示阶段，目标是尽快形成一个可运行的单聊 MVP。

开发内容：

- WebSocket 建连与鉴权
- 心跳与断线处理
- 单聊文本消息发送
- 消息持久化
- 在线投递
- 离线消息的最小回补能力

交付清单：

- `transport/ws` 初版结构
- `message` 模块最小闭环
- 文本消息 MVP
- 在线连接管理与投递入口
- 消息接口测试和最小集成测试

阶段后重构任务：

- 把 WebSocket 连接层与消息业务层解耦
- 让消息落库、消息分发、连接管理的职责更清楚
- 为后续会话摘要更新预留事件或调用边界
- 为 AI 消息和群消息预留统一消息抽象
- 为消息存储预留自动分表方案，按每 `10` 万条消息滚动到下一张物理表，并将分表路由收口在 repository / storage 层

完成标志：

- 单聊消息可以在在线和离线场景下跑通
- 项目第一次形成真正的“聊天”主链路

### Phase 3：会话

会话层负责承接消息列表和最近会话列表，是消息链路跑通后的关键整理阶段。

开发内容：

- 单聊会话创建或打开
- 最近会话列表
- 会话最后一条消息摘要
- 未读数
- 会话置顶、删除、清空未读等基础能力

交付清单：

- `conversation/session` 模型
- 单聊会话列表接口
- 会话摘要更新逻辑
- 未读数更新规则

阶段后重构任务：

- 把“会话视图”和“消息存储”职责分开
- 把会话摘要更新逻辑从 handler 中挪到应用层
- 为单聊与群聊会话抽公共接口
- 明确会话缓存策略和回源策略

完成标志：

- 用户打开聊天窗口、拉取会话列表已有清晰数据模型
- 消息模块可以直接写入会话摘要

### Phase 4：联系人与关系链路

这一步开始把 IM 中“人与人”的关系做起来，用来收紧消息权限和完善社交链路。

开发内容：

- 搜索用户
- 申请添加联系人
- 同意申请 / 拒绝申请
- 联系人列表
- 删除联系人
- 黑名单

交付清单：

- `contact` 关系表与申请表
- 联系申请状态流转
- 联系人列表查询接口
- 黑名单校验能力

阶段后重构任务：

- 抽出 `contact` 模块，避免把关系逻辑继续塞进 `user service`
- 给关系状态建立枚举和状态迁移规则
- 把缓存键、列表查询、状态变更统一收口
- 为消息发送时的“好友校验”预留接口

完成标志：

- 用户关系链路清楚可测
- 单聊消息链路能够按联系人关系增加权限控制

### Phase 5：群组

群组阶段用于把单聊能力扩展到多人场景。

开发内容：

- 创建群
- 邀请成员 / 移除成员
- 群成员列表
- 群角色与权限的最小版本
- 群会话
- 群消息

交付清单：

- `group` 模块
- 群成员关系模型
- 群消息投递与群会话更新
- 群资料查询与基础管理接口

阶段后重构任务：

- 将群权限校验从 handler 层下沉到应用层
- 梳理群会话与单聊会话的共性抽象
- 为群公告、群禁言、管理员等能力预留扩展点
- 统一联系人、群成员、消息投递中的用户查询入口

完成标志：

- 私聊和群聊两条主链路都已经具备
- 项目业务面足以支撑暑期实习项目展示

### Phase 6：AI 能力接入

这一阶段是必做项。目标是让 `Dipole` 具备区别于普通练手项目的 AI 能力，同时保证架构上可持续扩展。

接入前置能力与准备项，见 [ai-readiness-checklist.md](./ai-readiness-checklist.md)。

建议先做的 AI 功能：

- AI 助手账号
- AI 单聊会话
- 会话总结
- 智能回复建议
- 敏感内容或风险内容辅助判定

推荐模块结构：

- `internal/modules/ai/domain`
- `internal/modules/ai/application`
- `internal/modules/ai/infrastructure`
- `internal/modules/ai/delivery/http`

开发内容：

- 定义 AI 助手用户模型或助手标识体系
- 接入一个 LLM provider
- 设计 AI 请求上下文，能够读取最近消息和用户资料
- 设计异步任务或流式输出能力
- 将 AI 响应接入现有会话和消息模型

交付清单：

- 一个可用的 AI 助手 MVP
- 会话总结接口或定时总结任务
- 智能回复建议接口
- AI 调用日志、错误记录和基础限额控制

阶段后重构任务：

- 抽象 `provider` 接口，避免模型供应方耦合进业务层
- 把 prompt 模板、模型参数、系统指令配置化
- 把 AI 上下文拼装逻辑从 handler 中抽离
- 给 AI 消息单独定义元数据结构，避免污染普通文本消息
- 为后续多模型路由、工具调用、RAG 留好扩展点

完成标志：

- AI 能力能和消息、会话、用户体系自然协作
- 项目具备更强的差异化展示点

### Phase 7：Cgo 高性能模块接入

这一阶段也是必做项。目标是把“性能优化”从口号落成一个可解释、可 benchmark、可开关的工程模块。

推荐切入方式：

1. 先用 pure Go 跑通正确性
2. 用 benchmark 与 pprof 找出热点
3. 选择 1-2 个收益明显的点接入 Cgo
4. 保留 pure Go 回退实现

优先考虑的 Cgo 切入点：

- 文本敏感词匹配或文本预处理
- 消息压缩、编解码或序列化热点
- 大列表批处理时的高频字符串处理

推荐模块结构：

- `internal/platform/accelerator`
- `internal/platform/accelerator/purego`
- `internal/platform/accelerator/cgo`

开发内容：

- 为候选热点建立 benchmark
- 为 pure Go 和 Cgo 实现定义统一接口
- 使用 build tag 控制启用路径
- 加入边界测试、压力测试与回退策略

交付清单：

- 至少一个真正进入主流程的 Cgo 加速模块
- 对比 benchmark 数据
- 可切换的 pure Go / Cgo 实现
- 清晰的构建说明与依赖说明

阶段后重构任务：

- 收口内存管理边界，避免跨语言生命周期混乱
- 用更小的接口暴露 Cgo 能力，减少扩散
- 为 Cgo 模块补齐 fuzz test、race 检查和异常兜底
- 把性能测试脚本文档化

完成标志：

- Cgo 模块在项目里承担真实职责
- 性能收益可以通过数据解释
- 即使关闭 Cgo，系统也能稳定回退到 pure Go

### Phase 8：工程化补强

前面阶段把主功能做起来，这一阶段负责把项目打磨得更稳、更像完整系统。

开发内容：

- 监控指标
- 结构化日志
- 限流
- 后台管理的最小版本
- 压测脚本
- 部署脚本
- API 文档补全
- 评估并接入 `ants` 协程池，优先用于 WS 广播、批量事件投递等高并发短任务场景

交付清单：

- 健康检查、关键指标、错误日志
- 基础限流与防滥用能力
- 常用接口文档与消息链路说明
- 一套面向展示和自测的脚本

阶段后重构任务：

- 收口配置项，避免配置散落
- 统一中间件和错误处理路径
- 补齐观测维度，形成性能与稳定性的闭环
- 对关键链路做一次目录结构整理

完成标志：

- 项目具备更完整的展示面和维护性
- 后续继续加功能时，工程负担明显更小

## 5. 推荐的重构节奏

为了避免边做边乱，建议每个阶段结束后都固定做下面几类整理：

1. 补测试：把本阶段新增 service、handler、repository 的测试补齐。
2. 收边界：检查有没有把多个模块的逻辑堆进同一个 service。
3. 收 DTO：检查请求响应对象是否和数据库模型耦合过深。
4. 收目录：必要时把代码往 `modules / transport / platform` 目标结构上轻推一步。
5. 收文档：把新增接口、数据表和关键流程更新到 `docs/`。

---

## 6. 现在就需要预留的扩展点

虽然 AI 和 Cgo 会在后续阶段正式接入，但现在就应该提前留出位置：

- 用户模型里要允许出现系统用户或 AI 助手用户
- 消息模型里要允许挂载元数据，如 `source`、`extra`、`tool_result`
- 会话模型里要允许区分普通会话与 AI 会话
- 消息应用层要避免直接依赖具体 LLM SDK
- 文本处理、压缩、编解码等热点能力要通过接口调用，方便后续切到 Cgo 实现

---

## 7. 当前建议的最近三步

结合仓库现状，建议下一段开发按下面顺序推进：

1. 设计并落地 `transport/ws` 最小骨架，先完成 token 鉴权建连、在线连接管理和心跳。
2. 做单聊文本消息 MVP，跑通发送、落库、在线投递。
3. 在消息主链路初步稳定后，再补会话视图和联系人关系。

这样推进节奏会比较稳，既能持续交付，也能逐步把后续 AI 与 Cgo 的落点准备好。
