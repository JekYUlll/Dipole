# Dipole 缓存设计

## 1. 文档目的

这份文档用于明确 `Dipole` 的缓存分层、热点数据类型、失效策略与当前阶段的落地顺序。

目标有三点：

- 让 `Redis` 的职责清晰，不变成零散的临时 key 集合
- 让 IM 场景下的热点数据有统一缓存思路
- 控制当前阶段复杂度，先做最有收益的部分

---

## 2. 什么叫“多层缓存”

这里的“层”指的是**缓存所处的位置和职责层次**，不是简单把热点数据分几类。

对当前项目，更合适的三层缓存结构是：

1. 进程内缓存
2. Redis 共享缓存
3. 派生结果缓存

这三层分别解决不同问题。

---

## 3. 第一层：进程内缓存

### 3.1 作用

这一层位于单个 Go 进程内部，目标是减少同一实例内的重复查询和重复计算。

适合的数据特点：

- 读取频率高
- 数据体积小
- 允许极短时间的不一致
- 命中集中在单机请求路径里

### 3.2 适合缓存的内容

- 用户基础资料短缓存
- 群基础资料短缓存
- 配置项或常量映射
- 一些短时存在性判断结果

### 3.3 当前阶段实现

当前已经落地一项非常聚焦的进程内缓存：

- 用户 UUID 布隆过滤器
- 群组 UUID 布隆过滤器

作用是：

- 在高频接口里先过滤掉明显不存在的用户/群 UUID
- 减少无效请求落到 `Redis` 和 `MySQL`
- 给消息发送、好友申请、群操作这类路径做最轻量的前置过滤

当前策略：

- 服务启动时从数据库加载已有的用户 UUID 和群 UUID
- 新用户创建、新群创建时增量加入过滤器
- 过滤器只做“可能存在 / 一定不存在”判断
- 命中过滤器后，仍然走缓存或数据库做最终确认

这样实现的好处是：

- 代码改动小
- 几乎不增加业务耦合
- 对“无效 UUID”请求的收益很直接
- 不需要引入额外外部依赖

---

## 4. 第二层：Redis 共享缓存

### 4.1 作用

这一层是当前项目最应该优先建设的缓存层。

它负责：

- 跨请求复用热点数据
- 跨实例共享缓存结果
- 保存临时状态
- 承担高频读压力

### 4.2 适合缓存的内容

这一层是 IM 系统最常见的主缓存层，建议放这些数据：

- 用户资料
- 群资料
- 群成员列表
- 好友关系与黑名单关系
- 用户/群存在性判断
- 登录态附加状态
- 限流计数器
- 在线状态和连接节点信息

### 4.3 当前系统里 Redis 已经做了什么

当前已经在使用：

- `auth:revoked:<jti>`
- `user:profile:<uuid>`
- `group:meta:<group_uuid>`
- `group:members:<group_uuid>`
- `contact:relation:<user_uuid>:<target_uuid>`
- `rate:login:<telephone>`
- `rate:msg:<user_uuid>`
- `rate:file:<user_uuid>`
- `presence:online_users`
- `presence:online_connections`
- `presence:user:<user_uuid>:connections`

作用是：

- JWT 登出后的黑名单
- 让 token 在过期前也能立即失效
- 用户资料热点缓存
- 群资料与群成员热点缓存
- 联系关系与单聊权限热点缓存
- 登录、发消息、上传文件的短窗口限流计数
- 分布式在线用户数、在线连接数、多端连接状态缓存

对应代码在 [token_service.go](/home/horeb/_code/_go/Dipole/internal/service/token_service.go)。

### 4.4 这一层最值得优先落地的缓存

#### 用户资料缓存

建议 key：

- `user:profile:<uuid>`

缓存内容建议包含：

- `uuid`
- `nickname`
- `avatar`
- `status`
- `is_admin`

适用场景：

- 消息发送前校验目标用户
- 会话列表展示
- 联系人列表展示
- 群成员列表展示

建议策略：

- TTL：`5~15` 分钟
- 更新用户资料、状态时主动删 key

#### 群资料缓存

建议 key：

- `group:meta:<group_uuid>`

缓存内容建议包含：

- `uuid`
- `name`
- `notice`
- `avatar`
- `owner_uuid`
- `status`

适用场景：

- 拉群详情
- 群消息发送前权限判断
- 群广播事件推送

建议策略：

- TTL：`5~15` 分钟
- 更新群资料、解散群时主动删 key

#### 群成员列表缓存

建议 key：

- `group:members:<group_uuid>`

推荐存储结构：

- 当前阶段使用 Redis `Hash`

原因：

- 群成员除了 `user_uuid`，还带有 `role`、`joined_at` 等字段
- `Hash` 更适合按成员维度存取，也方便后续扩展群成员属性
- 读取整个群成员列表时，再在应用层做排序

适用场景：

- 群消息在线投递
- 群事件广播
- 群会话更新
- 群成员权限校验

建议策略：

- 加成员时可以增量写入，当前阶段先以“删缓存后重建”为主
- 移除成员/退群时删整个 key
- 解散群时删整个 key
- 如果本地和缓存出现不一致，允许回源 DB 后重建

#### 关系链缓存

建议 key：

- `contact:relation:<user_uuid>:<target_uuid>`

缓存内容建议表达：

- 是否好友
- 是否拉黑
- 备注名

适用场景：

- 私聊发消息权限判断
- 联系人列表和详情展示

建议策略：

- TTL：`5~10` 分钟
- 同意好友、删除好友、拉黑、取消拉黑后删 key

#### 限流计数缓存

建议 key：

- `rate:login:<telephone>`
- `rate:msg:<user_uuid>`
- `rate:file:<user_uuid>`

适用场景：

- 登录防刷
- 消息防刷
- 上传防刷

建议策略：

- 使用 `INCR + EXPIRE`
- 只保存短周期计数

#### 在线状态与多端连接缓存

建议 key：

- `presence:online_users`
- `presence:online_connections`
- `presence:user:<user_uuid>:connections`

当前实现方式：

- `presence:online_users` 使用 `ZSET`，score 为过期时间戳
- `presence:online_connections` 使用 `ZSET`，member 为连接 ID
- `presence:user:<user_uuid>:connections` 使用 `HASH`，field 为连接 ID，value 为连接元数据 JSON

当前缓存内容包含：

- `connection_id`
- `user_uuid`
- `node_id`
- `device`
- `device_id`
- `user_agent`
- `remote_addr`
- `connected_at`
- `last_seen_at`

适用场景：

- 后台查看在线用户数和连接数
- 多节点部署下共享在线状态
- 为后续多端同步、设备管理、强制下线预留基础

当前策略：

- `WebSocket` 建连时写入 Redis
- 收到客户端消息和服务端心跳时刷新 `last_seen_at`
- 断连时删除连接记录
- 额外依赖 TTL 清理异常断开后的残留状态

## 5. 第三层：派生结果缓存

### 5.1 作用

这一层缓存的不是原始业务数据，而是**由原始数据计算出来的结果**。

它适合：

- 计算成本较高
- 查询频率较高
- 允许短时间延迟更新

### 5.2 适合缓存的内容

- 后台概览统计
- 群消息已读统计
- AI 会话摘要
- 文件解析结果
- 搜索热点结果
- 风险判定结果

### 5.3 当前阶段建议

这层先不作为第一批代码落地重点。

原因是：

- 当前 IM 主链路还在补强
- 派生结果多数依赖更完整的消息/群/AI 链路
- 提前做容易变成过渡设计

---

## 6. IM 中最常见的热点内容

如果只看 IM 业务，最常见的热点通常集中在以下几类：

### 6.1 用户资料

原因：

- 消息、会话、联系人、群成员都依赖用户展示信息

典型字段：

- 昵称
- 头像
- 状态

### 6.2 群资料与群成员列表

原因：

- 群消息发送
- 群广播
- 权限判断
- 群详情展示

这通常是群聊系统里最热的一类读操作。

### 6.3 关系链

原因：

- 单聊权限判断
- 黑名单判断
- 联系人列表展示

### 6.4 会话首页数据

原因：

- 最近会话列表几乎每次打开客户端都要拉

典型内容：

- 最后一条消息摘要
- 未读数
- 排序信息

### 6.5 在线状态

原因：

- 连接管理
- 多端在线判断
- 消息投递路径选择

当前单机阶段可以先由内存 hub 维护，后续分布式时更适合迁到 Redis。

## 7. 推荐的缓存模式

当前项目建议以 `cache-aside` 为主。

也就是：

1. 先查缓存
2. 未命中时查数据库
3. 把结果写回缓存
4. 更新时先写数据库，再删缓存

这个模式最适合当前阶段，因为：

- 简单
- 可控
- 容易排查问题
- 不会把业务逻辑和缓存逻辑绑死

当前先不建议：

- write-through
- write-behind
- 复杂双写队列同步

---

## 8. 当前阶段的明确落地顺序

按收益和复杂度排序，建议这样推进：

1. 群成员列表缓存
2. 用户资料缓存
3. 群资料缓存
4. 会话首页热数据缓存
5. 已读/未读派生热数据

原因如下：

- 群成员列表缓存最能直接服务当前的群消息、群广播、群会话更新
- 用户和群资料缓存能立刻减少大量重复查库
- 关系链缓存能收紧单聊权限链路
- 限流计数和在线状态已经接入入口层，下一阶段更值得补会话热数据

---

## 9. 当前阶段不急着做的内容

以下内容先不作为第一批缓存开发目标：

- 本地进程内复杂缓存
- 多级缓存一致性
- 群消息已读人数缓存
- AI 摘要缓存
- 搜索结果缓存
- 向量检索缓存

这些能力更适合在 IM 主链路和 AI 主链路进一步稳定后再做。

---

## 10. 一句话结论

对于当前 `Dipole`，真正应该优先建设的是：

- 以 `Redis 共享缓存` 为核心的第二层缓存

其中最值得先落地的就是：

- `group:members:<group_uuid>`
- `user:profile:<uuid>`
- `group:meta:<group_uuid>`
- `contact:relation:<user_uuid>:<target_uuid>`
- `presence:user:<user_uuid>:connections`

这几项最符合 IM 的真实热点，也最贴近我们当前项目的业务重心。
